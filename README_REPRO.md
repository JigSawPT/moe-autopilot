# Reproduction guide

This document maps **every headline number to the exact command and evidence file it
came from**, so the results are checkable rather than asserted. All runs are on one
machine (RTX 5090 32 GB, driver 610.62 + Ryzen 9950X3D + 64 GB DDR5-6004 CL30 EXPO +
Windows 11 26200), fork branch built with VS2022 + `-T cuda=13.0`,
`CMAKE_CUDA_ARCHITECTURES=120`.

The full evidence bundle (the referenced `*.json` server-timing captures, the raw A/B
logs, the coverage tool, and the manifest) is described in [EVIDENCE.md](EVIDENCE.md).
The file names below are the ones in that bundle.

## 0. Build the fork

```bash
cmake -S . -B build -G "Visual Studio 17 2022" -A x64 -T cuda=13.0 \
  -DGGML_CUDA=ON -DCMAKE_CUDA_ARCHITECTURES=120 -DLLAMA_CURL=OFF
cmake --build build --config Release \
  --target llama-bench llama-server llama-cli llama-aipc-moe-profile -j 16
```

Linux/CUDA mainline equivalent: `cmake -S . -B build -DGGML_CUDA=ON -DLLAMA_CURL=OFF`
then build the same four targets. Windows CUDA-13.2/MSVC workaround:
[BUILD_WINDOWS.md](BUILD_WINDOWS.md).

## 1. Common protocol (how every number was measured)

- **Server:** `llama-server -m <gguf> -ngl 999 --n-cpu-moe <N> --no-mmap -c 8192 -t 16 --jinja`
- **Request:** a fixed NIF question (Portuguese), `temperature 0`, `max_tokens 400`,
  **2 repetitions**.
- **Metric:** the server's own `timings.predicted_per_second`.
- **Split ON:** set `AIPC_MOE_HOT_LIST=<hotlist>` and `AIPC_MOE_HOT_N=<n>` before launch.
- **Machine-state rules (mandatory):** `--no-mmap` (a cold NVMe cache drops throughput
  ~5×); `--poll 100` (default poll=50 costs 8–14%); and **working VRAM + system baseline
  ≤ ~30 GB** (above that, WDDM pages silently and *inverts* results — verify base VRAM
  with `nvidia-smi` before trusting any run).

**Non-circularity of the headline session number:** `session.hotlist` was built ONLY
from `session_history.txt` (3 tasks — IBAN, dates, a MoE explanation — generated by the
model itself). The evaluation task (NIF validator) does **not** appear in that history.
This is what makes the +26% an honest transfer number and not a memorized one.

## 2. Build a hot-list from representative text

```bash
# direct:
llama-aipc-moe-profile -m Coder-Next.gguf -f session_history.txt -ngl 999 --n-cpu-moe 24
mv aipc_moe_profile.hotlist session.hotlist

# or via the wrapper (also writes the counts json used for coverage):
python autopilot/autopilot.py hotlist --model Coder-Next.gguf \
  --from session_history.txt --out session.hotlist
```

The profiler writes two files: `aipc_moe_profile.hotlist` (the `"il id id ..."` text the
loader reads) and `aipc_moe_profile.json` (per-layer counts, consumed by the coverage
tool). `AIPC_MOE_HOT_LIST` points at the `.hotlist`.

## 3. Coverage (the x-axis of the ladder)

```bash
python autopilot/autopilot.py coverage --hotlist session.hotlist --profile live_profile.json --hot-n 96
```

`coverage` is the single canonical script: it reports the fraction of the live task's
expert hits (`live_profile.json`, the counts from the evaluation task itself) that fall
within the top-N of the hot-list. This is the number in the "coverage" column of every
table.

## 4. Primary model — Qwen3-Coder-Next MXFP4, `--n-cpu-moe 24`

Run the A/B (2b split ON vs 2a split OFF) for each hot-list and read
`timings.predicted_per_second`:

| Row in README | coverage | decode tok/s (2 reps) | evidence files |
|---|---|---|---|
| baseline, no split | 0% | 71.9 / 72.2 | `v2b_n28-nosplit_1.json`, `v2b_n28-nosplit_2.json` |
| generic corpus (`codernext.hotlist`) | 33.5% | 77.0 / 77.1 | `v2_hot96_1.json`, `v2_hot96_2.json` |
| **session (non-circular, `session.hotlist`)** | **68.6%** | **85.8 / 86.2** | `sess_eval_1.json`, `sess_eval_2.json` |
| ceiling (circular, `live.hotlist`) | 80.4% | 91.6 / 92.3 | `v2c_live_1.json`, `v2c_live_2.json` |
| post-review-fix regression check | 80.4% | 91.2 / 91.6 | `vfix_1.json`, `vfix_2.json` |

**On the 90.5–91.5 band in the README headline.** The raw non-circular A/B for the
session hot-list (the 68.6% row) reads **85.8 / 86.2 tok/s**. The **90.5–91.5 tok/s**
band quoted as the headline is the *same configuration* measured under a controlled,
freshly-low VRAM baseline (contemporaneous same-session control: best 90.93 / mean
90.53 tok/s at VRAM 31 680 MB). Both are real; the gap between them is exactly the
machine-state drift the discipline above exists to control. The MANIFEST is the source
of record and reports both — if you reproduce and land at ~86 with an elevated VRAM
baseline, that is expected, and pulling the baseline down recovers the ~90–91 band.

VRAM: baseline ~25 GB; `+HOT_N=96 ≈ +4.2 GB`.

## 6. Zero-overhead-when-disabled (parity check)

Built the branch, ran with **no AIPC env vars** and all experts in VRAM, so the split
code is never entered:

```bash
llama-bench -m Qwen3.6-35B-A3B-Q4_K_M.gguf -ngl 999 -ncmoe 0 -ub 2048 -mmp 0 -n 128 -r 3 -o json
# -> tg128 = 291.59 ± 0.71 tok/s ; no "AIPC" line in the output (split never entered)
```

Reference for parity: ~288.9 tok/s (upstream prebuilt) / 293.9 (a prior fork build).

## 7. The async-overlap negative result

The negative result is backed by direct A/B measurements under `v22_impl/` in the
evidence bundle (see [EVIDENCE.md](EVIDENCE.md)):

- **Marker split-boundary cost** (`PA_marker_off_*` vs `PA_marker_on_*`, anchored by the
  contemporaneous control `W6_control_*`): OFF mean ~90.5–90.7 vs ON mean ~82.8–83.3 =
  **−7 to −8 tok/s best-of ≈ +0.98 ms/token** (~41 µs per split-pair × 24 layers).
- **Scheduler-level async dispatch** (`P1_control_*` vs `P1_overlap_clean_*`): control
  ~90.7–90.9 vs overlap ~87.6–87.8 = **−3.4%**; the wait-time probe
  (`P1_waitdiag_*`) reports `already_done = 0.0%` over 9600 joins.

## 8. Prefill cost of the resident hot copies (three-arm A/B)

`llama-bench` on the primary model, identical flags in every arm, `pp2048` + `tg128`,
`-r 2`, the A/B run **twice** (independent model loads) to confirm reproducibility.
Note `-fa on` — this build's `llama-bench` takes `on|off|auto`.

```bash
# ARM A (control): no AIPC env vars
llama-bench -m Coder-Next.gguf -ngl 999 --n-cpu-moe 24 -b 2048 -ub 2048 --mmap 0 \
  -fa on -t 16 -p 2048 -n 128 -r 2 -o json

# ARM B (split armed): hot copies resident
AIPC_MOE_HOT_LIST=session.hotlist AIPC_MOE_HOT_N=96 \
llama-bench -m Coder-Next.gguf -ngl 999 --n-cpu-moe 24 -b 2048 -ub 2048 --mmap 0 \
  -fa on -t 16 -p 2048 -n 128 -r 2 -o json
# load log must show: "AIPC split: 72 hot tensors across 24 layers, 96 experts/layer, 4452 MiB of VRAM"

# ARM C (isolation): hot-list parsed but NO copies placed (split inactive)
AIPC_MOE_HOT_LIST=session.hotlist AIPC_MOE_HOT_N=0 \
llama-bench -m Coder-Next.gguf -ngl 999 --n-cpu-moe 24 -b 2048 -ub 2048 --mmap 0 \
  -fa on -t 16 -p 2048 -n 128 -r 2 -o json
```

| Metric | A mean | B mean | Δ (B vs A) | evidence files |
|---|---|---|---|---|
| pp2048 | 3154.1 | 2841.7 | **−9.90%** | `prefill_ab_A.json`, `prefill_ab_B.json` (run 1); `prefill_ab_A2.json`, `prefill_ab_B2.json` (run 2) |
| tg128 | 74.23 | 78.11 | **+5.22%** | same files |
| pp2048, ARM C | 3154.30 ± 15.00 | — | **+0.006% vs A** | `prefill_ab_C_hotn0.json` |

Per-run values: pp2048 A 3153.34 ± 13.32 / 3154.89 ± 16.72; B 2841.15 ± 13.54 /
2842.33 ± 3.60 — run 1 −9.90%, run 2 −9.91%, far outside the ~0.4% per-run stddev.
Load/split stderr logs: `prefill_ab_A.stderr`, `prefill_ab_B.stderr`; VRAM discipline
log: `prefill_ab_vram.log`. Full write-up: `prefill_ab.md`.

**Reading the three arms:** C == A to 0.006% proves parsing the hot-list is free and the
decode-only guard works (graph splits identical in all arms: 74 at bs=2048, 50 at bs=1).
The entire −9.9% is caused by the **4452 MiB of duplicate hot expert weights resident in
VRAM** in ARM B. Free VRAM stayed > 30 GB in both arms, so this is not WDDM paging.

**Instrument note:** the +5.22% tg128 is this llama-bench config's decode delta; the
headline +22–26% decode band comes from the *server-timings* instrument under the
serving config (sections 1 and 4). Different instruments and batch configs — do not mix
them; within this section both arms are measured identically.

**End-to-end consequence** (TTFT ≈ prompt/pp2048, total ≈ TTFT + output/tg128, using the
A/B means): chat 500+300 → split wins −183.3 ms; agent-loop 4000+100 → split loses
+72.5 ms; long-context 16000+300 → split loses +356.9 ms. **Break-even at
prompt:output ≈ 2.3.**

## 9. gpt-oss-120b reproduction (SWIGLU_OAI extension)

The base V2 split (sections 0-8 above) only covers the plain gated-SwiGLU MoE family
(Coder-Next). gpt-oss uses a different activation — a clamped gated-SiLU with a
per-expert bias (`SWIGLU_OAI`) — which the base split rejected and silently fell back
to the stock path for. The **`aipc-swiglu-oai`** branch (built off the published
`aipc-v2-pr` @ `ab2fbb7a6`) extends the dual hot/cold chain to also cover this family.
Full write-ups, correctness verification, and the config sweep are in
[`evidence/gpt-oss/`](evidence/gpt-oss/) (see its [README.md](evidence/gpt-oss/README.md)
for an index mapping each headline number to its file).

### Build

Same recipe as section 0, on the `aipc-swiglu-oai` branch instead of `aipc-v2-pr`:

```bash
cmake -S . -B build -G "Visual Studio 17 2022" -A x64 -T cuda=13.0 \
  -DGGML_CUDA=ON -DCMAKE_CUDA_ARCHITECTURES=120 -DLLAMA_CURL=OFF
cmake --build build --config Release \
  --target llama-bench llama-server llama-aipc-moe-profile -j 16
```

### Build a hot-list for gpt-oss (same tool as section 2, generic path)

```bash
llama-aipc-moe-profile -m gpt-oss-120b-mxfp4-00001-of-00003.gguf \
  -f <your-representative-text.txt> -ngl 999 --n-cpu-moe <N>
mv aipc_moe_profile.hotlist gptoss.hotlist
```

`<N>` (the `--n-cpu-moe` CPU-offload depth) is also the value to pass to `--n-cpu-moe`
in the bench/server commands below — the two must match, since the hot-list is built
against a specific set of CPU-resident (offload-eligible) layers.

### Headline command (llama-bench, the instrument used for every gpt-oss tok/s number)

```bash
# OFF (no cache — stock SWIGLU_OAI path)
llama-bench -m gpt-oss-120b-mxfp4-00001-of-00003.gguf \
  -ngl 999 -ncmoe <N> -b 4096 -ub 4096 -mmp 0 -t 16 -p 0 -n 128 -r 3 -fa 1 -o json

# ON (cache active)
# PowerShell: $env:AIPC_MOE_HOT_LIST="<path-to>/gptoss.hotlist"; $env:AIPC_MOE_HOT_N="<H>"
# bash:       AIPC_MOE_HOT_LIST=<path-to>/gptoss.hotlist AIPC_MOE_HOT_N=<H>
llama-bench -m gpt-oss-120b-mxfp4-00001-of-00003.gguf \
  -ngl 999 -ncmoe <N> -b 4096 -ub 4096 -mmp 0 -t 16 -p 0 -n 128 -r 3 -fa 1 -o json
```

Load-time stderr must show `AIPC split: … experts/layer, bias=yes, … MiB of VRAM` on
the ON arm to confirm the SWIGLU_OAI split is active (`bias=yes` — the tell that
distinguishes it from the plain-SwiGLU `bias=no` split in section 1). The best
(`<N>`, `<H>`) pairs measured on a 32 GB card: `ncmoe25/HOT_N22` (highest absolute
tok/s) and `ncmoe30/HOT_N42` (highest relative gain) — see
[`evidence/gpt-oss/gptoss_maxabs.md`](evidence/gpt-oss/gptoss_maxabs.md) and
[`evidence/gpt-oss/gptoss_sweep.md`](evidence/gpt-oss/gptoss_sweep.md) for the full
grid and the VRAM-budget math to pick `<H>` on other cards.

## Consistency note

Every number in the README and in this file is taken from `evidence/v2/MANIFEST.md`
(the source of record; the prefill three-arm A/B in section 8 is recorded there as a
dated addendum, with the full write-up in `prefill_ab.md`). Where a raw A/B value
(e.g. 85.8/86.2 for the non-circular session row) differs from a headline band
(90.5–91.5), both are stated and the difference is attributed to the documented
VRAM-baseline machine-state effect — the MANIFEST wins and reports both. The gpt-oss
numbers in section 9 and `evidence/gpt-oss/` are a separate, later evidence trail
(the SWIGLU_OAI extension) and are sourced from their own files rather than the V2
MANIFEST — each gpt-oss file documents its own method/hygiene inline.
