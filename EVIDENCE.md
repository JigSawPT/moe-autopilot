# Evidence bundle

Every headline number is backed by a raw server-timing capture. This file documents what
that evidence is, how it was produced, and which file proves which claim. The raw
`*.json` captures and per-run server logs are the reproducibility bundle referenced by
[README_REPRO.md](README_REPRO.md); the mapping below is the index into it.

## Protocol

Server `llama-server -m <gguf> -ngl 999 --n-cpu-moe <N> --no-mmap -c 8192 -t 16 --jinja`;
request = a fixed NIF question (PT) at `temperature 0`, `max_tokens 400`, 2 reps; metric =
`timings.predicted_per_second`. Split ON via `AIPC_MOE_HOT_LIST` + `AIPC_MOE_HOT_N`.
Hot-lists generated by `llama-aipc-moe-profile`; coverage computed by
`autopilot.py coverage` (single canonical script). See [README_REPRO.md](README_REPRO.md).

## Quality

At temperature 0 the split output is a correct, semantically-equivalent NIF validator
(same modulo-11 algorithm, PT-PT docstring) versus the clean binary; the post-fix
regression is within noise. Small stylistic divergence is the expected consequence of a
slightly different floating-point path.

## Notes on what is (and is not) in this public bundle

- The raw `*.json` captures are machine- and workload-specific traces. They are offered
  as reproducibility evidence, not as a dataset.
- Machine-specific hot-list artifacts (the actual `.hotlist` files and their counts
  JSONs for this author's workload) are **not** shipped — you generate your own with the
  profiler over your own representative text, as described in the quickstart.
