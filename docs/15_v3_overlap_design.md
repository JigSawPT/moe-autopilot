# 15 — V3 design: true async overlap of the cold CPU expert chain against GPU work

> **Status: DESIGN (no code, no GPU, no model runs in this pass).** This document designs the route that survived the async-overlap investigation (the "waves 4–6" negative result summarized in the project README): a *bespoke* fork/join that overlaps the RAM-resident (cold) MoE expert chain against CUDA work, **without** paying the scheduler/marker split-boundary tax that killed the scheduler-level and marker-split attempts. For the overlap route specifically it supersedes the earlier same-graph-scheduling framing; it does **not** re-open the CUDA-graphs route (refuted) or the scheduler-granularity / marker overlap (killed).
>
> All file:line anchors are against the patched llama.cpp fork (branch `aipc-v2`, HEAD `1e6630df2`) as read on 2026-07-04. Where this document cites `evidence/…/NOTES.md` and prior internal design notes, those are the author's working records that back the async-overlap negative result; the load-bearing measured facts are reproduced inline in §1.

---

## 0. The one-paragraph thesis

Waves 5 and 6 proved that overlap **at scheduler granularity is impossible** and overlap **enabled by a graph marker is uneconomical** — but neither tested the actual hypothesis. The scheduler executes splits in strict index order and the CPU backend blocks the scheduler thread inside `ggml_graph_compute` (`ggml-cpu.cpp:170-190`), so at every cold→hot boundary the GPU stream is already drained (wave 5: `already_done = 0/9600`, ~181 µs block/join). The marker route tried to carve the atomic CUDA region so an async join could land at the merge, but a split boundary between two CUDA regions **requires** a non-CUDA node (`ggml-backend.cpp:1303` cuts a split on every backend change), costing ~41 µs/pair × 24 (wave 6: −8.3%). **V3's bet: neither tax is fundamental to overlap — both are artifacts of routing overlap through `ggml_backend_sched`.** If we keep ONE graph but replace the sched's serial `compute_splits` loop with a bespoke executor that (a) dispatches the cold split of layer L on a persistent worker thread the instant its inputs are ready, (b) keeps issuing the *remaining CUDA splits* (this layer's hot+merge, and critically **layer L+1's attention**) on the CUDA stream without blocking, and (c) enforces the merge's dependency on the cold result with a **CPU-completion → CUDA-stream-wait event bridge** instead of a host block — then the per-layer sync cost is a std::atomic store + a `cudaStreamWaitEvent` (~1–5 µs), the GPU never drains, and `min(cold 181 µs, hot 96 µs) ≈ 96 µs/layer` is hidden. That is the ~+2.3 ms/token prize → ~100–104 tok/s. **Spike 1 measures the bare fork/join cost in isolation before any model work; if it exceeds 20 µs/layer, the route cannot net-win and we stop.**

## 2. What the code actually gives us (the primitives V3 will compose)

Read this section as "the toolbox". Every claim is a file:line.

### 2.1 The scheduler is serial by construction — and that is the *only* reason overlap fails today
`ggml_backend_sched_compute_splits` (`ggml-backend.cpp:1541`) loops `for split_id in 0..n_splits` (`:1549`) and for each split: copies inputs (`:1555`), then dispatches `ggml_backend_graph_compute_async(split_backend, &split->graph)` (`:1678`). For a CUDA split that call is genuinely async (returns after enqueue). **For a CPU split it is fully synchronous** — `ggml_backend_graph_compute_async` → `iface.graph_compute` → `ggml_backend_cpu_graph_compute` → `ggml_graph_compute` (`ggml-cpu.cpp:170-190`), which blocks the caller until the whole cold chain finishes. The CPU backend exposes **no async, no events** (`.set_tensor_async=NULL … .synchronize=NULL … .event_record=NULL`, `ggml-cpu.cpp:193-206`; device `.async=false`, `:396`; `.event_*=NULL`, `:496`). **Consequence:** the scheduler thread cannot both run the cold chain and keep issuing CUDA work — it is *inside* the cold chain. This is the load-bearing fact. V3 must take the cold chain off the scheduler thread onto our own worker so the issuing thread stays free.

### 2.2 The merge's dependency is a device copy, enforced by a host sync today
The cold result (`node_94`, the `down` output) is consumed by the merge `SUB` (node 108) as `CUDA0#node_94#0` — a device copy of the cold tensor into a CUDA-resident input buffer. In the sched loop, that copy is the split's input-copy step (`:1555-1675`); for the non-user-input path it calls `ggml_backend_synchronize(input_backend)` where `input_backend` = CPU (`:1588` fast path, `:1665`/`:1669` general path). Because CPU `synchronize` is `NULL`, the "sync" is implicit: the CPU `graph_compute` already returned, so `node_94` is populated before the copy runs. **The merge dependency is therefore: (cold done) → H2D copy of node_94 → merge kernels read the copy.** V3 keeps exactly this dependency but changes *how* "cold done" is signalled to the GPU: from "scheduler thread already blocked past it" to "worker atomically flags done; a `cudaStreamWaitEvent` gates the merge". The math is untouched → exactness preserved (§5).

### 2.3 CUDA has the full event fork/join toolkit already
- `ggml_backend_cuda_event_record` = `cudaEventRecord(event, cuda_ctx->stream())` (`ggml-cuda.cu:4610-4613`).
- `ggml_backend_cuda_event_wait` = `cudaStreamWaitEvent(cuda_ctx->stream(), event, 0)` (`:4616-4620`) — a **GPU-side** wait: the host does not block; the stream waits on the event. Exactly the merge-gate primitive V3 needs.
- **`cudaLaunchHostFunc` bridge exists in-tree** (`:4629`, currently `#if 0`) — enqueues a host callback on a CUDA stream. This is the CPU→CUDA direction: a host function that spins on the worker's atomic and returns, letting the stream proceed. Candidate for the merge-gate if we want the *stream* to wait on the *CPU* without a manager-thread block (see §3, Candidate C's bridge variant).
- **8 non-blocking CUDA streams per device**, lazily created (`common.cuh:178 GGML_CUDA_MAX_STREAMS=8`, `:1462-1467` `cudaStreamCreateWithFlags(..., cudaStreamNonBlocking)`). The sched path uses stream 0 (`curr_stream_no=0`); V3 can keep everything on stream 0 (no cross-stream complexity) since the CPU cold chain is the parallel branch, not a second GPU stream.
- **CUDA graphs are keyed by first-node-ptr** with multiple graphs per context (`common.cuh:1401-1429`) explicitly "when the computation is split across CPU/GPU (e.g., with `--n-cpu-moe`)". So a two-cgraph or one-graph-many-splits design still gets independent capture per CUDA subgraph — no conflict with the wave-4 fact that the big region is already captured.

### 2.4 There is already a GPU-side concurrent fork/join — but it is GPU-streams-only
The fork ships `ggml_cuda_concurrent_event` (`common.cuh:1240-1379`) and drives it in `ggml_cuda_graph_evaluate_and_capture` (`ggml-cuda.cu:4320-4449`): at a `fork_node` it records `fork_event` on the main stream and makes N sub-streams `cudaStreamWaitEvent` it (`:4341-4346`); at the `join_node` it records each sub-stream's event and makes the main stream wait (`:4423-4430`). Population is a graph-optimize pass (`:4720-4840`) gated to `attn_norm` fan-out only (`:4725`). It even does a **write-range overlap safety check** (`common.cuh:1281-1379`, `is_valid()`). **Relevance to V3:**
- It proves the CUDA event fork/join plumbing works and is trusted in this fork.
- It is **NOT** a CPU‖GPU primitive — every branch is `stream_mapping[node] = stream_id ≥ 1`, i.e. all branches are CUDA sub-streams of the same device. The RAM cold chain cannot be a branch (it's a CPU graph, not CUDA nodes). So V3 cannot simply "add the cold chain as a concurrent branch". **But** the `is_valid()` write-range analysis is a reusable model for V3's thread-safety audit (§4): it already reasons about which tensors two concurrent producers may write.
- **Escalation-worthy nuance (not a contradiction):** this mechanism shows the fork's maintainers already accept intra-graph concurrency with event joins. V3's CPU‖GPU fork/join is the natural CPU-side sibling. It does not contradict any measured fact.

### 2.5 The CPU threadpool is a busy-spin at `--poll 100` — the core contention risk
`ggml_graph_compute_poll_for_work` (`ggml-cpu.c:3128-3141`) spins `ggml_thread_cpu_relax()` (`_mm_pause` on x86, `:518`) for `n_rounds = 1024*128*poll = ~13.4M` iterations between graphs before falling back to a cond-var sleep (`:3151-3157`). At `--poll 100` (mandatory per docs/11) the 16 worker threads **busy-wait on cores** whenever they're between splits. Wave 5's diagnosis noted the worker's "per-token OpenMP-team handoff (~a few µs × 24)" made it a slight net loss even ignoring the zero overlap. **V3's worker must not fight this spin** — the affinity plan (§4.4) is not optional; it is the difference between overlap and cache-line war on the 9950X3D.

### 2.6 The orchestration seam
`llama_context::graph_compute` (`llama-context.cpp:2425-2452`) is the single call site: it sets the threadpool on the CPU backend (`:2431-2437`), sets n_threads on all backends (`:2440-2442`), then calls `ggml_backend_sched_graph_compute_async(sched, gf)` (`:2444`). `process_ubatch` (`:1304-1364`) builds/allocs the graph and calls `graph_compute` (`:1364`). **This is where V3 diverts:** when `AIPC_OVERLAP=1` and the graph carries the AIPC split signature, call our bespoke executor instead of (or wrapping) `ggml_backend_sched_graph_compute_async`.

## 3. Design Question 1 — where the fork/join lives (three candidates, graded)

### Candidate A — two persistent cgraphs, orchestrated in llama-context with per-layer sync
**Shape.** At build time, partition the decode graph into `g_gpu` (attention + aipc-prep + hot chain + merge + shared experts + sampling, all CUDA) and `g_cold` (the 24 CPU cold-chain segments). In `process_ubatch`, launch `g_cold` on a persistent worker via `ggml_backend_graph_compute(cpu_backend, g_cold)` while the main thread drives `g_gpu` on CUDA. Per-layer, the merge of layer L waits on cold_L.

**What synchronizes.** A per-layer event pair: worker signals cold_L done (atomic + optionally a `cudaLaunchHostFunc` that records a CUDA event); the merge_L kernels are gated by `cudaStreamWaitEvent`. The H2D copy of cold_L is issued by the worker (or a small manager) once cold_L is done.

**Where the merge dependency is enforced.** At merge_L: `cudaStreamWaitEvent(stream0, cold_done_event_L)` before the merge `SUB` kernel. Because the cold result must be H2D-copied first, the copy must complete before the event fires (copy on stream0 after a host-side wait on the atomic, or copy issued by the worker on a dedicated copy stream that the merge waits on).

**Per-layer sync cost estimate.** 1 atomic store (worker) + 1 `cudaStreamWaitEvent` (manager) + 1 small H2D (cold result, ~n_embd×k floats ≈ tens of KB, already paid today) ≈ **2–6 µs/layer** IF the H2D can be overlapped. **Risk: graph partitioning.** Two separate cgraphs mean two separate `ggml_backend_sched_alloc_graph` allocations, two CUDA-graph capture keys, and — the hard part — **the activation buffer produced by `g_gpu`'s aipc-prep (the `cur` input to the cold chain) and consumed by `g_cold` must live in a buffer both graphs see, with correct lifetime across the graph boundary.** The current single-graph flow gets this for free (one allocator, one context). Splitting it re-introduces exactly the cross-graph buffer lifetime problem doc-14 §M3b-i flagged as "high risk" (`llama-context.cpp:122` note on `work_data` realloc). **Grade: viable but heaviest; the two-allocator/two-context seam is where bugs and non-determinism hide.**

### Candidate B — custom split executor, ONE graph, bypass `ggml_backend_sched_compute_splits`
**Shape.** Keep the existing single graph and the existing sched *split plan* (the `sched->splits[]` array, built by `ggml_backend_sched_split_graph`, `:1014-1404`). Replace only the *execution* loop `ggml_backend_sched_compute_splits` (`:1541-1725`) with our own `aipc_compute_splits_overlap` that walks the same `splits[]` but, when it recognizes the **cold CPU split of layer L** (signature: `split->backend_id == CPU` AND `nodes[0].op == MUL_MAT_ID` reading a HOST-WEIGHTS buffer AND output `ne[2] ≤ 8` = decode guard — exactly wave 5's proven signature), dispatches that split's `ggml_backend_graph_compute` on a **persistent std::thread** and *continues the loop* issuing the subsequent CUDA splits. It joins the worker (atomic spin or futex) only at the split whose inputs reference the cold split's output nodes — i.e. the merge — and gates that split's first CUDA kernel with a `cudaStreamWaitEvent` on a CPU-completion event.

**Why this is strictly better than wave 5.** Wave 5 *also* used a worker thread in this loop — but it joined at the **immediately following split** (`:1656-1675`, the H2D input-copy of node_94), because that split's input-copy `must see node_94 populated`. Wave 5 could not push CUDA work *before* that join because the very next split's first action reads the cold result. **V3's difference is the join point and what runs before it:** V3 must arrange for **layer L's hot chain + layer L+1's attention** to be *issued to the CUDA stream* before the join, so the GPU has work in flight while cold_L runs. The blocker (wave 5 root cause) is that attention_{L+1} lives *after* merge_L in the split order, inside a split that depends on node_94. **So Candidate B alone does not solve it either — unless we also reorder the split *issue* order** (issue the hot-chain split and the L+1 attention split *before* the merge split, deferring only the merge). That reorder is legal in a bespoke executor because we control issue order; the *data* dependency is still enforced by the `cudaStreamWaitEvent` at the merge. **This is the crux insight of V3: the executor issues in dependency order, not split-index order, and defers the merge behind an event.**

**What synchronizes.** `std::atomic<uint32_t> cold_epoch` per layer (worker stores L+1 when cold_L done); the manager issues CUDA splits; before the merge kernel it does a lightweight host wait on the atomic (`while(cold_epoch.load(acquire) <= L) _mm_pause();`) *then* the H2D copy *then* the merge — OR records a CUDA event via `cudaLaunchHostFunc` so the stream self-gates (no manager block). No sched marker, no extra split boundary → **no 41 µs tax** (that tax was the *graph-structure* split-count increase 50→98; here `n_splits` is unchanged, we only change execution order and add an atomic).

**Where the merge dependency is enforced.** At the merge split's input-copy: the cold result's H2D copy is issued only after `cold_epoch > L`; the merge kernel runs after that copy on the same stream (natural stream ordering) — identical data path to today, so identical numerics.

**Per-layer sync cost estimate.** 1 atomic store + 1 atomic-acquire spin (bounded: if cold already done, ~0; if not, we *want* to have hidden it behind GPU work) + the H2D (unchanged) ≈ **1–5 µs/layer** of pure glue. **Grade: chosen candidate (see §7).** It reuses the sched's split-plan machinery (no graph partitioning, one allocator, one context, one CUDA-graph key set), touches exactly one function, and its risk surface is the reorder + the join semantics — both testable in isolation (Spike 2).

### Candidate C — the primitives found, and why they don't replace B (but augment it)
Three in-tree mechanisms were evaluated as alternatives:
1. **`ggml_cuda_concurrent_event`** (§2.4): GPU-streams-only; cannot host the CPU cold chain as a branch. **Reject as the fork/join carrier**, but *reuse its `is_valid()` write-range model* for the §4 aliasing audit.
2. **`cudaLaunchHostFunc` bridge** (`:4629`, `#if 0`): lets a CUDA *stream* wait on a *host* condition without a manager-thread block — the stream calls a host fn that spins on `cold_epoch`. **Adopt as the merge-gate variant inside Candidate B** if the manager-thread host-wait proves to add latency; it moves the wait onto the stream. Caveat: host callbacks on a captured CUDA graph are restricted (a host func inside a graph capture is legal via `cudaGraphAddHostNode` but the current capture path `:4602` uses stream capture — a host func during capture is allowed but executes at *launch*, not capture; must verify it doesn't break the wave-4 capture of the big region). **Spike 2 tests both merge-gate variants.**
3. **`ggml_backend_sched` async APIs** (`ggml_backend_sched_graph_compute_async`, `:1889`): these just call `compute_splits` (`:1901`) — the serial loop. No hidden async capability. **Reject.**

**Verdict on Q1:** the fork/join lives in a **bespoke single-graph split executor (Candidate B)** that issues CUDA splits in dependency order, runs the cold split on a persistent worker, and gates the merge with a CPU→CUDA event (host-spin default, `cudaLaunchHostFunc` variant as fallback). Candidate A is the fallback IF the reorder within one graph proves to violate an allocator/aliasing invariant that only a hard graph partition can satisfy (escalation trigger, §8).

## 4. Design Question 3 — thread-safety audit (two backends computing concurrently)

Concrete checklist, each item with the code fact and the V3 mitigation.

### 4.1 Shared ggml_context / graph node array
The single decode graph and its `ggml_context` are read-only during compute (built in `process_ubatch` before compute, `:1337`). Both the worker (cold split) and the manager (CUDA splits) only **read** node metadata and **write** to *disjoint* tensor data buffers (cold writes node_94's data; CUDA writes hot/merge buffers). **Risk:** the sched's `compute_splits` mutates shared scratch (`ids`, `used_ids` vectors at `:1546-1547`, `prev_ids_tensor` at `:1545`) — V3's executor must give the worker its **own** copy of any such scratch, or ensure the worker's cold split needs none (the cold `mul_mat_id` reads ids already materialized by aipc-prep on the GPU and copied down — verify the cold split's ids source is stable before dispatch). **Mitigation:** worker touches only `ggml_backend_graph_compute(cpu_backend, &cold_split_graph_view)`; no shared sched scratch. Audit the input-copy of the cold split's ids (must be completed by the manager before the worker is dispatched, since it's an H2D... actually D2H for cold ids — the ids come from the GPU aipc-prep and must be copied to CPU before the cold `mul_mat_id` runs; **the manager must issue+sync that ids copy before releasing the worker**). This is a real ordering constraint — see §4.6.

### 4.2 Allocator ownership
Single graph ⇒ single `ggml_gallocr` / sched buffer allocation (`ggml_backend_sched_alloc_graph`, `llama-context.cpp:1347`). No concurrent allocation. **Risk:** none for allocation *timing* (alloc happens before compute). **Risk that IS real:** the sched packs split I/O copies into a shared graph buffer sized `graph->n_nodes + n_splits*MAX_SPLIT_INPUTS*2*n_copies` (`:1393`). The cold split's output copy region (`CUDA0#node_94#0`) and the CUDA splits' regions are distinct slots — verify (via the `is_valid()`-style range check) that the worker's cold output tensor and any concurrently-running CUDA kernel's output do **not** alias in that buffer. **Mitigation:** at build/first-run, run a one-time overlap check (reuse `common.cuh:1281` logic) over {cold split output tensors} × {CUDA splits issued before the join} and **abort overlap (fall back to serial) if any range overlaps.** This is both a safety net and the exactness guarantee.

### 4.3 CUDA stream per graph
All CUDA splits stay on **stream 0** (`curr_stream_no=0`, the sched's stream). The worker runs pure CPU (no CUDA). The only cross-device ops are: (a) D2H copy of ids/cur into the cold split's CPU inputs (before worker dispatch), (b) H2D copy of node_94 (after worker done, before merge). Both are already in the graph. **No new stream needed** — avoids the wave-4/6 complications entirely. **Risk:** the H2D of node_94 must be ordered *after* the worker finishes writing node_94 (CPU) — a CUDA async copy issued while the CPU is still writing would race. **Mitigation:** the H2D is issued by the manager only after the atomic `cold_epoch > L` (host-visible completion of the CPU write), OR by the `cudaLaunchHostFunc` gate. Never issue the H2D speculatively.

### 4.4 Threadpool contention with `--poll 100` — the 9950X3D CCD plan
This is the highest-severity thread-safety item because it's a *performance* correctness issue, not a crash. Facts:
- The main CPU threadpool (16 threads, `-t 16`) busy-spins at `_mm_pause` between splits (`ggml-cpu.c:3128-3141`, `n_rounds ~13.4M` at poll 100).
- The cold split's `ggml_backend_graph_compute` uses **that same threadpool** (the CPU backend's threadpool, set in `graph_compute` at `llama-context.cpp:2435`). So "the worker" dispatching the cold split actually kicks the *shared* 16-thread pool.

**This is a design fork.** Two sub-options:
- **B1 — cold split uses the shared main threadpool (16 threads).** The worker std::thread just calls `ggml_backend_graph_compute`, which kicks all 16. During cold_L, the GPU is issuing CUDA kernels from the *manager* thread — but the manager is one of those 16 (or a 17th). If the manager is thread 0 of the pool, it's *inside* the kickoff barrier and cannot issue CUDA. **This breaks unless the manager is a separate thread that does not participate in the CPU threadpool.** So B1 requires: manager = the calling thread (llama-context's decode thread), cold worker = a dedicated std::thread that kicks the 16-thread pool, and the 16-thread pool must not include the manager's core.
- **B2 — cold split uses a *dedicated* smaller threadpool pinned to one CCD.** Create a second `ggml_threadpool` for the cold chain (e.g. 8 threads on CCD1) via `ggml_threadpool_new` (exported, `ggml-cpu.cpp:668`), set it on a second CPU backend instance or swap it in for the cold split. The main GPU-issuing thread + its light CPU work stays on CCD0. **This is cleaner** — no shared-pool barrier ambiguity — at the cost of a second threadpool and possibly fewer threads for cold (measure: cold is memory-bandwidth-bound at high offload, so 8 threads may saturate RAM anyway).

**9950X3D specifics (from memory + directive):** 2 CCDs × 8 cores. **CCD1 measured better for decode** (directive). Plan: **manager/GPU-issue thread + the main non-expert CPU work pinned to CCD0; the cold expert threadpool pinned to CCD1** (8 threads, the fast-decode CCD, with the 3D V-Cache). The `--poll 100` spin then happens on CCD1 cores that are *doing the cold work*, not fighting the manager. Affinity via `ggml_thread_apply_affinity` + `cpumask` (`ggml-cpu.c:507`, `:2163`). **Kill/verify in Spike 3:** if pinning doesn't separate them, the spin-war reappears as the wave-5 "few µs × 24 net loss". **Recommendation: design for B2 (dedicated CCD1 cold pool) but Spike 1 measures B1 vs B2 fork/join cost so we pick with data.**

### 4.5 KV cache and activation buffer aliasing between concurrent work
KV cache is written by attention (GPU), read by attention. The cold chain (experts) does **not** touch KV — it reads `cur` (post-attention activation) and writes expert output. **Risk:** if V3 issues attention_{L+1} (which writes KV_{L+1}) concurrently with cold_L (which reads cur_L), do they alias? cur_L is a distinct activation buffer from KV. **But** the graph allocator may *reuse* buffer regions across the layer boundary (activation buffers are recycled aggressively). **Mitigation:** the §4.2 one-time range check must include cur_L's buffer vs any buffer written by the concurrently-issued L+1 CUDA splits. If the allocator recycles cur_L's region for an L+1 tensor that a concurrent kernel writes, **that's the aliasing bug** — abort overlap for that layer or force cur_L to a non-recycled buffer (pin via `ggml_backend_sched_set_tensor_backend` won't help; may need a dedicated persistent buffer for the cold input, which is a design cost). **This is the #1 correctness risk and the #1 reason to keep the join *tight* (issue only this layer's hot+merge before the join, defer L+1 attention) as the conservative first prototype (Spike 2), then widen to L+1 lookahead (Spike 3) only after the range check proves it safe.**

### 4.6 The D2H ordering constraint (ids/cur must reach the CPU before the worker runs)
The cold `mul_mat_id` needs (a) `cur` and (b) `sel_cold` ids on the CPU. Both are produced on the GPU (attention output, aipc-prep). Today the sched copies them D2H as the cold split's inputs (`:1555-1675`) **before** dispatching the cold split — and because the copy path syncs, they're ready. **V3 must preserve this:** the manager issues the cold split's input D2H copies **and synchronizes them** (or waits on their event) **before** releasing the worker. Only *then* is the worker dispatched. **This serializes "GPU produces cur" → "D2H" → "cold runs" — which is correct and unavoidable; the overlap is cold_L against the *next* GPU work (hot_L, L+1 attention), not against cur_L's own production.** This matches the physics: you can't run the expert on an activation that doesn't exist yet.

**Summary of §4:** the audit yields a **conservative first prototype** (tight join: cold_L ‖ hot_L only, no L+1 lookahead, dedicated CCD1 cold pool, one-time range-check guard that falls back to serial on any aliasing) and an **aggressive target** (cold_L ‖ {hot_L, attention_{L+1}} once the range check clears L+1). Spike 2 builds the conservative one; Spike 3 attempts the aggressive one.

## 5. Design Question 4 — exactness plan (temp-0 logit identity)

**The math is untouched.** V3 changes *scheduling and synchronization*, not a single tensor op: the cold chain, hot chain, and merge (`experts = cold + (hot−cold)*mask`, `llama-graph.cpp:1879-1881`) are byte-for-byte the same nodes, same order, same buffers. There is **no approximation, no stale-merge, no dropped dependency** — the merge still waits for the *current* cold_L (the atomic/event gate enforces "cold_L fully done" before the merge reads it). This is the strongest possible exactness position: identical graph, identical numerics, only the *thread* that runs the cold split and the *mechanism* that signals its completion change.

**Proof protocol (reuse existing harness, no new methodology):**
1. **A/B logit identity** via `evidence/v2/v22_impl/run_ab.ps1`: run the fixed NIF prompt at `temperature 0`, `max_tokens 400`, 2 reps, with `AIPC_OVERLAP=0` (control) and `AIPC_OVERLAP=1` (treatment). The harness already captures `choices[0].message.content` and checks `def validar_nif` + mod-11 (`run_ab.ps1:112-115`).
2. **Byte-diff the generated text**: control vs treatment output must be **character-identical** at temp 0 (greedy decode is deterministic given identical logits). Any single-token divergence = a stale/raced merge = **fail, revert** (this is the kill for Spike 2/3).
3. **Stronger: logit-level diff** (optional, if text-diff is ever ambiguous): dump the first-token logit vector for both arms (the fork has `copy_tensor_async_floats` for logits, `llama-context.cpp:1993`) and assert max-abs-diff == 0. Text-identity is the primary gate; logit-diff is the tiebreaker.

**Determinism caveat list (things that could make control≠treatment even with correct code):**
- **CUDA atomics / reduction order:** the merge math (`SUB/MUL/ADD`) and the `mul_mat_id` kernels are **unchanged** — same kernels, same launch config, same reduction order. V3 does not touch kernel internals, so any non-determinism present in control is *equally* present in treatment → they still match each other (the gate is control==treatment, not control==some-reference). **This is safe as long as V3 does not change the CUDA-graph capture boundary of the merge** (which would change fusion/reduction). Candidate B keeps `n_splits` unchanged → capture boundary unchanged → safe. (Contrast wave 6, which changed capture — though even there graph *reuse* stayed identical, `398` both arms.)
- **H2D/D2H copy timing:** copies are size- and content-identical; timing doesn't affect values.
- **Threadpool thread count for cold:** if Spike 3 uses a *different* thread count for the cold pool (B2, 8 threads on CCD1) than control (16), the CPU `mul_mat_id` reduction is **tree-reduced per-thread** — **does thread count change the reduction order?** `ggml_barrier` + `current_chunk` chunking (`ggml-cpu.c:1444`, `:1698`) partitions rows across threads; the final accumulation per output element is within one thread's chunk (mul_mat_id output row is owned by one thread), so **thread count should not change per-element float order** for mul_mat. **VERIFY in Spike 2** by A/B'ing control-16-threads vs a serial-8-thread cold run at temp 0 *before* adding overlap — if 16-thread and 8-thread cold produce identical logits, B2's dedicated pool is exactness-safe. **If not, pin the cold pool to the same thread count as control (16 on CCD1, or keep B1).** This is a concrete Spike-2 sub-test.

**Exactness verdict:** the design is exactness-preserving by construction (identical graph/math), and the gate is a byte-diff of temp-0 output between overlap-off and overlap-on. The single caveat requiring active verification is cold-pool thread-count vs reduction order (Spike 2 sub-test).

## 6. Design Question 2 & 5 — the spike ladder with kill criteria and effort

### Spike 1 — bare fork/join microbenchmark (no model) — **THE decisive first test**
**Goal:** measure the realistic per-layer cost of the exact fork/join V3 will use, in isolation, before touching the model. This directly tests the wave-6 hypothesis: *was the 41 µs/pair a SCHED artifact, and is a bespoke fork/join ~1–5 µs?*

**Design (toy harness, standalone .cpp linked against ggml + ggml-cuda, ~150 LOC):**
- Build a tiny ggml graph: one CPU `mul_mat` (size ≈ the cold chain's per-layer work, e.g. `[2048×2048] × [2048×10]` in MXFP4 or F32 — tuned so it takes ~180 µs on the CCD1 pool) **and** one CUDA `mul_mat` (~96 µs). Or, even simpler for the *pure glue* number: no matmuls, just `std::thread` + `std::atomic` + `cudaEvent` ping-pong.
- **Three measurements, each × 24 (layers) × 1000 (iterations):**
  1. **Pure glue:** manager thread does `atomic.store(i)`; worker spins `atomic.load()==i`, does trivial work, stores done; manager `cudaEventRecord` + `cudaStreamWaitEvent` on a dummy stream. Measure wall time / (24×1000). **This is the floor of V3's per-layer sync cost.**
  2. **Realistic fork/join:** manager issues a real ~96 µs CUDA `mul_mat` on stream0; concurrently the worker runs a real ~180 µs CPU `mul_mat` on the CCD1 pool; manager gates a second CUDA kernel (the "merge") with a `cudaStreamWaitEvent` on the worker-completion event (via `cudaLaunchHostFunc` spinning on the atomic, or a host-wait-then-record). Measure the *added* time vs running them serially. **Expected: ≈ max(180,96) + glue vs 180+96 serial → the overlap saves ~96 µs/layer minus glue.**
  3. **B1 vs B2 pool contention:** run measurement 2 with the CPU matmul on (B1) the same 16-thread pool the manager's spin would use vs (B2) a dedicated 8-thread CCD1 pool, with the manager thread `_mm_pause`-spinning on CCD0 to simulate `--poll 100`. **Measures whether the spin-war (wave-5's net loss) is real and whether CCD pinning fixes it.**

**KILL CRITERION:** if the realistic fork/join (measurement 2) costs **> 20 µs/layer** of glue (i.e. the overlap saving is < 76 µs/layer after glue), **the route cannot net-win — STOP.** Rationale: at 20 µs/layer glue × 24 = 0.48 ms/token overhead, against a max saving of `min(96,181)×24 = 2.30 ms` best-case; but real overlap is partial and the cold tail (181−96=85 µs) stays exposed, so glue must be a small fraction of the 96 µs window. >20 µs/layer eats too much and re-creates the wave-6 economics.
**Secondary read:** if measurement 1 (pure glue) alone is >20 µs, the primitive itself (atomic + event + host-func) is too heavy on this Windows/WDDM/Blackwell stack → the route is dead regardless of model integration. This is the cheapest possible kill.
**Effort:** **1–2 days** (standalone harness, no fork changes, no model load). Highest information-per-hour in the whole program.

### Spike 2 — single-layer prototype in the fork behind `AIPC_OVERLAP=1`
**Goal:** overlap cold_L ‖ hot_L for the offloaded layers using Candidate B's bespoke executor, **tight join only** (no L+1 lookahead), dedicated CCD1 cold pool, range-check guard.
**Build:** replace `ggml_backend_sched_compute_splits` call for the AIPC decode graph with `aipc_compute_splits_overlap` (env-gated; falls back to the stock loop when off or when the range-check fails). Persistent worker std::thread created once. The reorder is minimal: for each offloaded layer, dispatch cold split on worker → issue hot-chain split on stream0 → at merge split, host-spin on `cold_epoch` → H2D → merge. **No L+1 reorder yet** (conservative, §4.5).
**Exactness sub-tests (before measuring speed):** (a) 16-thread vs 8-thread cold reduction-order A/B (§5 caveat); (b) temp-0 text byte-diff overlap-off vs overlap-on.
**KILL CRITERION:** if per-layer time (or total tok/s) is **> serial baseline** (i.e. overlap-on is slower than overlap-off in the same session), the tight-join overlap doesn't even break even → the hot window (96 µs) is too small to hide meaningful cold behind, OR the glue is worse than Spike 1 predicted → **stop and reassess** (likely means the real win needs L+1 lookahead, which is Spike 3's harder bet; decide whether to attempt it or kill the program). Also kill immediately on any temp-0 text divergence.
**Effort:** **1–2 weeks** (the executor, the worker lifecycle, the CCD pinning, the range-check guard, the two exactness sub-tests). This is where most of the fork engineering lives.

### Spike 3 — full 24-layer + L+1 lookahead + A/B
**Goal:** widen the join so cold_L overlaps {hot_L, attention_{L+1}} — hiding cold behind the *larger* GPU block (attention is bigger than the 96 µs hot chain), which is where the real headroom is (§1 asymmetry note). Requires the §4.5 range-check to clear cur_L vs L+1 buffers; if it doesn't, either force a persistent cold-input buffer or stay at tight-join and accept the smaller win.
**A/B:** full `run_ab.ps1` protocol, NIF prompt, non-circular session hot-list, `--poll 100`, VRAM ≤ 30 GB logged, temp-0 byte-identity, 2 reps.
**CONTINUE CRITERION:** ≥ **97 tok/s** (clearly outside the 90.5–91.5 control band + machine-state noise) to justify continuing to productionization. **TARGET: ~100–104 tok/s.**
**KILL:** if < 97 after both tight-join and L+1-lookahead are exhausted, or if L+1 lookahead cannot be made exactness-safe (aliasing), **the program lands at whatever tight-join delivered; if that's inside the control band, V3 is a negative result — document and stop** (matching the wave-4/5/6 discipline: a measured negative is a valid outcome).
**Effort:** **1–2 weeks** (the L+1 reorder, the wider range-check, full A/B + 35B direction check).

### Spike 4 — productionize + 35B arm + adversarial review (method rule)
**Goal:** only if Spike 3 ≥ 97 (ideally ~100–104). Harden the worker lifecycle (clean shutdown, no stale threads), the fallback path (any unrecognized graph → stock sched loop), the CCD affinity as a tunable, and the range-check as a permanent guard. Re-run the **Qwen3.6-35B-A3B arm** (`--n-cpu-moe 16`, HOT_N=48) to confirm the gain moves the same direction (expect a *smaller* relative gain on the faster model — §7). Then **ultracode adversarial review** of the concurrency (race conditions, the atomic memory-ordering, the aliasing guard, the exactness argument) **before any commit** (method rule: no premise-invalid or unreviewed concurrency lands).
**Effort:** **1 week** (hardening + 35B + review). Commit only after review passes and A/B holds.

**Total effort ladder:** 1–2 d (Spike 1) → 1–2 wk (Spike 2) → 1–2 wk (Spike 3) → 1 wk (Spike 4) ≈ **4–7 weeks** if every gate passes, but **the program can die at 1–2 days (Spike 1 kill) for near-zero sunk cost** — which is the point of leading with the microbench.

## 7. Design Question 6 — upstream-mergeability

**Would this land as a llama.cpp PR?** The *general form* is attractive to upstream and plausibly mergeable; the AIPC-specific hot/cold split is not.

- **The general primitive:** "async offload-overlap for MoE — run the CPU-resident (offloaded) expert `mul_mat_id` on a worker thread concurrently with the GPU work of the same and next layer, joining at the point the GPU consumes the CPU result." This is **not AIPC-specific** — it applies to *any* llama.cpp run with `--n-cpu-moe` / `-ot` putting expert tensors in RAM (the mainstream MoE-offload case: any user running a big MoE partially on CPU). The bespoke executor (Candidate B) is exactly the kind of thing upstream might accept as an opt-in `--moe-overlap` flag, *if* it's clean and generalizes beyond the hot/cold dual path.
- **Why the current form is fork-only:** V3 as prototyped keys off the AIPC cold-split *signature* (host-weights `mul_mat_id`, decode guard) and the hot/cold merge structure. Upstream has no hot/cold split. The **general** PR would drop the hot-copy entirely and just overlap the single CPU expert chain against the GPU attention/next-layer — simpler than AIPC's dual path.
- **What the general PR looks like:** a new execution mode in `ggml_backend_sched` (or a `ggml_backend_sched_compute_splits_overlapped` variant) that, given a split plan, detects CPU splits whose *only* consumer is a later GPU split, dispatches them on a persistent worker, and gates the consumer with a CPU-completion → `cudaStreamWaitEvent` bridge (the `cudaLaunchHostFunc` at `:4629`, which upstream left `#if 0` — V3 would be the motivating use case to finish it). The write-range aliasing guard (`common.cuh:1281` model) becomes the safety gate. **This is a genuinely general improvement to CPU-offload MoE decode throughput** — the community-impact angle the user cares about.
- **Sequencing for impact:** prove it in the fork (Spikes 1–4) with AIPC's measured numbers, then extract the general `--moe-overlap` form as the upstream PR, citing the fork's A/B as evidence. The concurrent-stream work already in this fork (§2.4) shows upstream is receptive to intra-decode concurrency with event joins — V3's CPU-side sibling is a natural next contribution. **Recommendation: design Candidate B's executor from the start with a clean `is CPU-split-with-only-GPU-consumers?` predicate (not an AIPC-signature predicate) so the general form is a small refactor, not a rewrite.**

## 8. Design Question 7 — honest regime analysis (where overlap pays vs where it can't)

**Overlap pays when the per-layer floor is dispatch/latency-dominated, not bandwidth-dominated.**

- **Where it WINS — the incumbent (Qwen3-Coder-Next, 512 experts, 10 used, MXFP4, `--n-cpu-moe 24`, 68.6% coverage).** Here cold ≈181 µs/layer is a *mix* of RAM-gather bandwidth (only 31.4% of experts, ~132 KB/layer... wait: cold reads the 31.4% *cold* fraction) and per-layer dispatch/threadpool-wake overhead. Critically the GPU side (hot 96 µs + attention) is *latency/launch-bound* (wave-4: X already graph-captured, small). Because both sides have a fixed per-layer component and the GPU has real work to run during cold, overlap hides ~96 µs/layer → the modeled **91 → ~100–104 tok/s**. This is the design's home turf.
- **Where it CANNOT help much — RAM-bandwidth-bound MoE (gpt-oss-120b class, ~42 GB/s effective gather).** If the cold chain is *pure* RAM bandwidth (all experts cold, huge per-layer byte traffic, DDR5 saturated at ~42 GB/s), then cold time is `bytes / 42 GB/s` and is **not** hideable-for-free: overlapping it behind GPU work still requires the GPU work to be *longer* than the cold RAM read to fully hide it, and if the model is offload-heavy the GPU per-layer work is *small* relative to the massive RAM read. **Honest estimate: single-digit % gain.** Concretely: if cold_L = 500 µs (bandwidth-bound) and GPU_L (hot+attention) = 150 µs, overlap hides only 150 µs of the 500 → cold tail 350 µs still fully serial → the layer drops from 650 µs to 500 µs = 23% *on that layer*, but if the layer is 90% of the token time and the token was already RAM-bound, the *token* gain is bounded by `hidden_GPU_time / token_time` = `24×150µs / total` — and on a bandwidth-bound token the GPU time is a small slice, so **~3–8% token-level gain, likely single-digit.** The RAM bandwidth is the wall; overlap cannot move a wall, only hide latency behind it.
- **The discriminator:** overlap's ceiling per layer is `min(GPU_work_L, cold_L)` hidden. When `GPU_work_L ≥ cold_L` (fast MoE, moderate offload — incumbent), you hide *all* of cold → big win. When `cold_L >> GPU_work_L` (bandwidth-bound, heavy offload — gpt-oss-120b), you hide only `GPU_work_L` (small) and the cold tail dominates → small win. **This is why the incumbent at 91→~103 is the right target and gpt-oss-120b is explicitly a "measure, expect single-digit" case — do not oversell it.**

**Design implication:** V3 is a **latency-hiding** optimization, not a bandwidth optimization. It should ship as an opt-in that *helps* dispatch-floor-dominated configs and is *neutral* (falls back to serial cleanly) for bandwidth-bound ones. The range-check guard + serial fallback already give it the "do no harm" property for the regimes it can't help.

## 9. Escalation register (per directive — report, don't improvise)

Reviewed every measured fact against the code; **no contradiction found.** The code reading *confirms* the wave-5/6 diagnoses precisely (serial `compute_splits`, blocking CPU backend, atomic CUDA region, split-on-backend-change at `:1303`). Items to flag:

1. **The merge dependency CAN be enforced without a sched boundary** — via `cudaStreamWaitEvent` on a CPU-completion event within a bespoke executor (Candidate B), keeping `n_splits` unchanged. This is the whole bet; if Spike 2 shows the event/atomic gate *cannot* be inserted without the sched re-cutting a split (e.g. because the H2D copy of node_94 unavoidably forces a split-index boundary the executor can't reorder around), **escalate** — that would mean no candidate can enforce the dependency without the tax, and the route is truly closed. *Current code reading says the executor controls issue order and the copy is an input-copy step we can defer, so this is expected to hold — but it is the load-bearing assumption to verify first in Spike 2.*
2. **cur_L vs L+1 buffer aliasing (§4.5)** is the correctness cliff for the aggressive (L+1 lookahead) target. If the allocator recycles cur_L's region into an L+1 tensor a concurrent kernel writes, and it can't be cheaply pinned, the aggressive target is unreachable and the program lands at the smaller tight-join win. **Escalate at Spike 3** if L+1 lookahead can't be made safe *and* tight-join is inside the control band (negative result).
3. **cudaLaunchHostFunc during CUDA-graph capture** (§3 Candidate C): the merge-gate-via-host-func variant must not break the wave-4 capture of the 133-node region. If it does, fall back to the manager-host-spin variant (which doesn't touch capture). Not an escalation — a designed fallback.

## 10. Summary (the decision this document commits to)

- **Chosen architecture: Candidate B — a bespoke single-graph split executor** (`aipc_compute_splits_overlap`, env-gated `AIPC_OVERLAP=1`) that reuses the sched split-*plan* but replaces the serial `compute_splits` execution: cold split → persistent CCD1 worker; issue remaining CUDA splits in **dependency order** (hot_L, then L+1 attention in the aggressive target); gate the merge with an atomic + `cudaStreamWaitEvent` bridge — **no marker, no extra split, `n_splits` unchanged, no 41 µs tax.** Fallback to Candidate A (hard two-cgraph partition) only if the reorder violates an allocator/aliasing invariant a single graph can't satisfy.
- **Why B over A:** one allocator, one context, one CUDA-graph key set, one function touched; the two-context/two-allocator seam of A is where non-determinism and lifetime bugs live. B also extracts cleanly to the general upstream `--moe-overlap` PR.
- **Spike 1 (decisive, 1–2 days, near-zero sunk cost):** standalone microbench of the bare fork/join (pure-glue floor + realistic CPU-matmul ‖ CUDA-matmul + event join + B1-vs-B2 CCD contention). **KILL if realistic glue > 20 µs/layer.** This tests the wave-6 "was the tax fundamental?" hypothesis before any model work.
- **Two biggest risks:** (1) **cur_L ↔ L+1 activation-buffer aliasing** (§4.5) — the correctness cliff that may cap the win at the smaller tight-join level; (2) **`--poll 100` spin-war on the 9950X3D** (§4.4) — the wave-5 net-loss mechanism, mitigated only by correct CCD1 pinning of a dedicated cold pool, which must be proven in Spike 1 (B1 vs B2) and Spike 3.
- **Effort ladder:** Spike 1 (1–2 d, kill >20 µs/layer) → Spike 2 (1–2 wk, kill if slower than serial or any temp-0 divergence) → Spike 3 (1–2 wk, continue ≥97 tok/s, target ~100–104) → Spike 4 (1 wk, productionize + 35B + ultracode review before commit). **≈4–7 weeks fully, but dies cheaply at day 2 if the primitive is too heavy.**
- **Exactness:** identical graph and math; gate = temp-0 output byte-identity (overlap-off vs overlap-on) via `run_ab.ps1`; one active caveat (cold-pool thread-count vs reduction order) verified in Spike 2.
- **Honest scope:** a *latency-hiding* win for dispatch-floor-dominated MoEs (incumbent 91→~103); *single-digit %* and "measure, don't oversell" for RAM-bandwidth-bound MoEs (gpt-oss-120b). Ships opt-in with a serial fallback so it does no harm where it can't help.
