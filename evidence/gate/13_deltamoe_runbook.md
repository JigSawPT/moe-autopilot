Contamination rule (implemented in the script): an example is rejected if any blacklist line has a word-level Jaccard similarity > 0.6 with the example's prompt.

## F3.A — Distillation datasets

**Format (JSONL, one per line):**
```json
{"domain":"codigo-pt","teacher":"coder-next@73d5d804f","messages":[{"role":"user","content":"..."},{"role":"assistant","content":"<teacher's response>"}],"meta":{"temp":0.7,"max_tokens":600,"ts":"2026-07-05"}}
```
- **Splits:** by prompt hash: `sha1(prompt) % 100` → 0-89 train, 90-94 val, 95-99 test. The test set is NEVER seen in training or checkpoint selection.
- **Allowed prompt sources:** variations generated by the teacher from 30-50 manual seeds per domain (ask "generate 20 realistic variations of this task, in PT-PT"); our own technical docs; NEVER the blacklist.
- **Teachers (served by the fork, MANIFEST config):** `codigo-pt` (code-pt) → Coder-Next (86 tok/s); `pt-tecnico-clinico` (pt-technical-clinical) and `raciocinio` (reasoning) → gpt-oss-120b (33 tok/s, `reasoning_effort: low` via chat_template_kwargs for direct answers).
- **Cheap pre-generation gate (F3.A kill):** 30 domain prompts, teacher vs base 27B side by side, judged by gpt-oss as a blind arbiter (1-5 rubric). If the teacher doesn't win by ≥0.7 points on average, there's no delta worth training for — kill the domain.
- v0 targets: 2,000 examples/domain (code ≈ 2.5 h; gpt-oss ≈ 7 h/domain, overnight).

## F3.C — Domain benchmarks (benchmark_lab format; 20/domain)

File: `deltamoe/bench_<domain>.py` in the SAME schema as `benchmark_lab.py` (required/forbidden/regex → deterministic). **8 concrete examples per domain** (expand to 20 following the pattern; objective checks mandatory; prompts disjoint from the blacklist AND from the datasets):

**codigo-pt (code-pt):** (1) PT postal code validator `NNNN-NNN` [required: `re.`, regex \d{4}-\d{3}]; (2) CSV parser with quoting [required: `csv.`|`split` + test case]; (3) retry decorator with backoff [required: `def retry`, `time.sleep`]; (4) function computing VAT at 6/13/23% [correct FINAL regex on a given case]; (5) refactor a given function into a list comprehension [forbidden: `for i in range(len`]; (6) SQL: total per customer, last 90 days [required: GROUP BY, forbidden: SELECT *]; (7) PowerShell moving logs with -LiteralPath [forbidden: Remove-Item]; (8) debug a given off-by-one [required: line identification + fix].

**pt-tecnico-clinico (pt-technical-clinical):** (1) explain HbA1c to a diabetic patient [required_any: glycated hemoglobin; forbidden: definitive diagnosis]; (2) leaflet summary (given text) in 3 bullets [must_include facts from the text]; (3) structured referral letter [required: reason, history]; (4) EN→PT-PT translation of a technical abstract [required: European target terms, forbidden: test Brazilianisms: "infarto","derrame"]; (5) explain heart attack (enfarte) vs stroke (AVC) [required: enfarte, AVC]; (6) posology: convert mg/kg to a dose (arithmetic verifiable by FINAL regex]; (7) pre-AO90: rewrite a given paragraph [required: "acção"|"óptimo" per the text — pre-AO90 orthography, kept in Portuguese by design]; (8) triage: chest pain → required: 112 (European emergency number).

**raciocinio (reasoning):** (1-8) in the style of the gate's tasks but NEW: syllogisms with FINAL=yes/no; time/percentage arithmetic with FINAL=value; scheduling with windows; interpreting a given benchmark; logical ordering of steps; contradiction detection in a text; counting with a trap; unit conversion — all with a deterministic FINAL regex.

**Runner:** `python deltamoe/run_domain_bench.py --profile <profile> --dominio X` → same CSV/jsonl as the lab. Comparisons: base | base+LoRA(domain) | teacher. **Promotion**: LoRA ≥ +15% vs base in the domain AND ≥ base−1 on the 22-task gate AND decode ≥90% of base.

## F3.D — Router (contract)

- **Form:** FastAPI proxy on `127.0.0.1:18300/v1/chat/completions` (OpenAI-compatible; clients change nothing).
- **Server startup:** all adapters loaded at scale 0: `--lora lora_codigo.gguf --lora lora_pt.gguf --lora lora_raciocinio.gguf`.
- **Per request:** classify → `POST http://127.0.0.1:18201/lora-adapters` with `[{"id":0,"scale":1.0},{"id":1,"scale":0.0},{"id":2,"scale":0.0}]` → forward the request → (optional) reset scales to 0.
- **v0 classifier:** rules (```/def/SELECT → code; clinical/pre-AO90 terms → pt; FINAL=/how much/logical-why → reasoning); v0.1: embeddings (small local model) + logistic regression trained on the datasets themselves (domain field).
- **Fallback:** confidence < threshold → all scales 0 (pure base). **Logging:** JSONL per request `{ts, predicted_domain, confidence, adapter_used, tok_s, n_tokens}` — this log is what feeds F3.E (uncertainty) and hot-list re-profiling.
- **Acceptance:** 50 blind mixed requests: router ≥95% of manual-selection quality; overhead <50 ms/request.

**Diagnosis — rubric saturation, not teacher weakness.** With the student at 4.20-4.83/5 on these broad, easy prompts, the ≥0.7 mean-delta criterion is arithmetically impossible in raciocinio (headroom 0.17) and demands a near-perfect teacher elsewhere (Claude scored 27/30 fives in codigo-pt and still missed the bar). Two consecutive teacher pools (local, frontier) "failed" a gate that cannot mathematically pass at these student baselines. **Method rule (promoted):** before gating on a bounded-rubric mean delta, verify headroom (rubric_max − student_mean) ≥ threshold; if not, gate on failure-set pass rates instead.

**Secondary finding (thesis-relevant, honest):** under blind local judging with deterministic anchors, the clean dense 27B is near-indistinguishable from frontier Claude on broad everyday PT-PT tasks at this difficulty — both sides ace what they complete. The distillable frontier gap must therefore live where the student *fails*: harder, longer-horizon, workload-realistic tasks. That is what F3.A.2 measures.

## F3.A.2 — Failure-set gate (design; launched 2026-07-04)

Replaces the saturated rubric gate. Teacher pool unchanged (frontier Claude; optional Codex second arm).

1. **Hard-pool authoring (no GPU):** 40 prompts/domain, realistic and workload-flavoured, difficulty targeted at plausible 27B failure modes (multi-constraint code with edge-case traps and a concrete test input; long-form clinical documents with strict PT-PT/pre-AO90 + mandatory structure + posology arithmetic; multi-step quantitative/logic with compound constraints). EVERY prompt ships an objective, machine-checkable verification (required/forbidden regex and/or an expected `FINAL=` value) and the expected values are **computed by executing a reference solution**, never hand-derived. Disjointness enforced at authoring time: word-level Jaccard ≤ 0.6 vs the 22-task gate blacklist, the 90 run-1 cheap-gate prompts, and the F3.C `bench_*.py` tasks. Hard-set prompts join the blacklist: they NEVER enter training data verbatim (dataset prompts must be fresh variations, as before).
2. **Student pass (GPU):** dense 27B, temp 0, `max_tokens 1500`; deterministic scoring → the **failure set** = items with failed checks.
3. **Teacher pass (no GPU):** frontier Claude answers the failure set only; scored by the SAME deterministic checks. Prompt-author bias (Claude authors, Claude teaches) is mitigated by the objective checks and the realism requirement — items the teacher also fails simply don't count toward viability.
4. **Viability criterion (per domain):** ≥ 8/40 student failures (enough signal to train toward) AND teacher passes ≥ 70% of the student-failed items. If NO domain reaches 8 student failures at realistic-hard difficulty, the honest conclusion is that the 27B has no broad distillable gap on this machine, and DeltaMoE pivots (smaller student class, or ultra-narrow micro-domains mined from real workload traces).
